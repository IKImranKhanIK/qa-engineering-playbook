# Contributing to QA Engineering Playbook

Thank you for your interest in contributing. This guide outlines how to contribute effectively.

## How to Contribute

### Reporting Issues
- Search existing issues before creating a new one
- Use the appropriate issue template
- Provide clear, specific descriptions
- Include examples when relevant

### Suggesting Content
- Check the [ROADMAP.md](ROADMAP.md) for planned topics
- Open an issue with the `content-request` label
- Explain the value and target audience
- Reference real-world scenarios when possible

### Submitting Changes

#### Before You Start
1. Fork the repository
2. Create a branch from `main`
3. Use a descriptive branch name: `feature/topic-name` or `fix/issue-description`

#### Making Changes
1. Follow the existing structure and format
2. Ensure consistency with existing content
3. Test any code examples
4. Update the table of contents if adding new files

#### Pull Request Process
1. Ensure your PR has a clear title and description
2. Reference any related issues
3. Wait for review and address feedback
4. Squash commits if requested

## Content Guidelines

### Writing Style
- Clear and concise
- Engineering-focused
- No fluff or filler content
- Avoid motivational language
- Use technical precision
- Define acronyms on first use

### Structure
Each lesson should include:
- Concept explanation
- Why it matters in real engineering environments
- Common mistakes or pitfalls
- Real-world examples
- Hands-on exercise or lab reference
- Senior-level insights

### Code Examples
- Must be complete and functional
- Include comments for complex logic
- Use realistic variable names
- Show both correct and incorrect approaches where helpful

### Templates
- Must be production-ready
- Include instructions for use
- Provide example sections with placeholders
- Explain each section's purpose

### Labs
- Clear objectives
- Step-by-step instructions
- Expected outcomes
- Evaluation criteria
- Difficulty level indicator

## Technical Requirements

### Markdown
- Use GitHub-flavored Markdown
- Follow consistent heading hierarchy
- Use code fences with language specifiers
- Keep line length reasonable for readability

### File Naming
- Use lowercase with hyphens: `test-design-techniques.md`
- Number lesson files: `01-topic-name.md`
- Use descriptive names that indicate content

### Directory Organization
```
docs/XX-module-name/
├── README.md          # Module overview and navigation
├── 01-first-topic.md
├── 02-second-topic.md
└── assets/            # Images or diagrams if needed
```

## Quality Standards

### Content Accuracy
- Verify technical details
- Reference authoritative sources
- Update outdated information
- Correct any errors promptly

### Completeness
- Cover the topic thoroughly
- Address common questions
- Include relevant edge cases
- Provide context for decisions

### Practical Value
- Focus on actionable information
- Include real-world applications
- Avoid purely theoretical content
- Demonstrate proven practices

## Review Process

### What Reviewers Check
- Technical accuracy
- Content completeness
- Writing clarity
- Structural consistency
- Link validity
- Code functionality

### Response Time
- Initial review: within 7 days
- Follow-up reviews: within 3 days
- Complex changes may require additional time

## Recognition

Contributors will be:
- Listed in commit history
- Mentioned in release notes for significant contributions
- Acknowledged in the repository

## Questions

If you have questions about contributing:
- Check existing issues and discussions
- Review this guide thoroughly
- Open an issue with the `question` label
